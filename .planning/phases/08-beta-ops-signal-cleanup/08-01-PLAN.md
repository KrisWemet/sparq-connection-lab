# Plan 08-01 — Remove False-Positive Journey Start Error Signal

Wave: 1

## Objective

Investigate why `beta_primary_path_error` with `stage = journey_start_lookup` is being written on successful runs, and narrow or remove that false-positive signal without changing the primary user flow.

## Scope

- inspect the current `journey_start_lookup` logging path in the supported beta flow
- identify whether the tracked value represents a real failure, an expected empty lookup, or a logging-shape bug
- implement the smallest fix that preserves true error detection while removing the false-positive write

## Expected Changes

- focused server-side journey-start error logging changes
- tighter distinction between expected control flow and real errors
- no user-facing flow changes

## Explicitly Out Of Scope

- new analytics events
- dashboard, onboarding, or daily-growth redesign
- broad observability changes
- secondary fallback path work

## Verification

- `npm run lint`
- focused local verification around the journey-start path
- production deployment of the narrow logging fix

## Success Criteria

- a successful primary-path journey start no longer writes the false-positive `beta_primary_path_error` row
- real journey-start failures would still be trackable after the fix
