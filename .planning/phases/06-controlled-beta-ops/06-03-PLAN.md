# Plan 06-03 — Error Monitoring And Beta Ops Guardrails

Wave: 2

Depends on:
- `06-01`
- `06-02`

## Objective

Add the minimum error monitoring and logging needed to support the proven primary path during controlled beta, while explicitly keeping the secondary login-entry fallback path as a follow-up hardening item.

## Scope

- add structured error monitoring or logging for the primary signup-driven path
- capture errors around signup, onboarding, journey selection, dashboard arrival, and Day 1 start/completion
- document exactly what operators should watch during controlled beta
- explicitly mark the secondary login-entry fallback path as follow-up rather than expanding this phase

## Expected Changes

- structured error logging hooks or monitoring integration for the primary path
- operator notes for triage and evidence review
- planning and status updates that keep the fallback path out of the active beta gate

## Explicitly Out Of Scope

- broad observability platform rollout
- redesigning error UX
- stabilizing the secondary login-entry fallback path in this phase

## Verification

- `npm run lint`
- focused tests for primary-path error capture where practical
- controlled manual verification that failures on the primary path are visible to operators

## Success Criteria

- primary-path errors are visible enough to support real beta users
- operators have clear evidence sources for funnel health and user feedback
- the secondary login-entry fallback path remains an explicit follow-up hardening item, not hidden scope
