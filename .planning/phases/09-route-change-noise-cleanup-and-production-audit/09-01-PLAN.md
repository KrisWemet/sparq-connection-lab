# Plan 09-01 — Remove False-Positive Route Change Error Noise

Wave: 1

## Objective

Investigate and narrow the `beta_primary_path_error` rows caused by `route_change` during register redirects, without changing the primary user flow.

## Scope

- inspect the current `routeChangeError` reporting path in the supported beta flow
- identify whether the logged `Route Cancelled` cases are expected redirect churn or real failures
- implement the smallest fix that preserves useful route-error reporting while suppressing expected redirect noise

## Expected Changes

- focused client-side route error filtering or narrowing
- no user-facing redirect or flow changes
- no broad observability redesign

## Explicitly Out Of Scope

- new analytics events
- onboarding, dashboard, or daily-growth redesign
- broad cleanup of unrelated client errors
- secondary fallback-path stabilization

## Verification

- `npm run lint`
- focused local verification around the route-change reporting path
- production deployment of the narrow logging fix

## Success Criteria

- expected register redirect churn no longer writes false-positive `route_change` beta-path errors
- real route failures would still be reportable after the fix
