# 15-02 Plan

Phase: 15
Wave: 2
Title: Production Fail-Soft And Analytics Verification

## Objective

Prove in live production that the playful slice remains fail-soft and that its analytics are visible, distinguishable, and non-confusing relative to the core funnel.

## Scope

- verify production behavior if the playful endpoint is unavailable or failing
- confirm dashboard still loads when playful prompts are unavailable
- confirm daily-growth still loads when playful prompts are unavailable
- confirm the serious core remains unchanged during playful outage conditions
- verify playful analytics are visible and distinguishable from the core funnel

## Tasks

1. create or use a safe production verification method to simulate or observe playful endpoint unavailability without endangering the main path
2. verify dashboard behavior under playful endpoint failure
3. verify daily-growth home behavior under playful endpoint failure
4. confirm the main primary-path funnel events still emit correctly
5. confirm playful analytics events are present and clearly separable from the core funnel events
6. record exact pass or fail results in the live evidence output

## Acceptance Criteria

- dashboard still loads when the playful endpoint is unavailable
- daily-growth still loads when the playful endpoint is unavailable
- the serious core CTA and daily loop remain unchanged during playful outage behavior
- playful analytics events are queryable and separate from the primary-path funnel events
- no new production blocker is introduced by the playful slice

## Verification

- targeted production browser proof for playful outage behavior
- targeted `analytics_events` verification for:
  - `playful_daily_spark_viewed`
  - `playful_daily_spark_tried`
  - `playful_daily_spark_swapped`
  - `playful_daily_spark_sent`
  - `playful_favorite_us_viewed`
  - `playful_favorite_us_saved`
  - `playful_favorite_us_sent`
  - primary signup-driven funnel events on the same or adjacent run

## Notes

- prefer narrow, reversible verification methods
- do not widen this work into a new outage framework or broad observability expansion
