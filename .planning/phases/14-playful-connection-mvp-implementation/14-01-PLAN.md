# 14-01 Plan

Wave: 1
Goal: Add the contained data and delivery layer for the playful MVP without touching critical onboarding or core daily-loop state.

## Milestone Plan

Build the backend and state foundation for:
- `Daily Spark`
- `Favorite Us`

This wave should keep new logic isolated and non-blocking.

## Files Likely Affected

- `src/pages/api/**` for a small playful-content delivery endpoint
- `src/lib/server/**` for prompt selection and analytics helpers
- `src/lib/**` for client fetch helpers or feature gating
- `src/services/analyticsService.ts`
- data source files for prompt definitions, likely under `src/data/**`
- optional migration or schema docs under `supabase/**`

## Data Model Needs

- curated prompt library with fields such as:
  - `id`
  - `type` (`daily_spark`, `favorite_us`)
  - `category`
  - `solo_ok`
  - `partner_optional`
  - `tone`
  - `energy`
  - `active`
- per-user interaction state:
  - `user_id`
  - `prompt_id`
  - `seen_at`
  - `swapped_at`
  - `acted_at`
  - `sent_at`
  - optional `saved_response`
- isolated analytics events for:
  - viewed
  - swapped
  - tried
  - sent
  - saved

## Acceptance Criteria

- the playful layer can return one prompt for the dashboard context
- the layer can return one prompt for the positive reflection context
- all fetches fail safely without blocking dashboard or daily-growth load
- no changes to onboarding routing, auth flow, or core daily session state
- analytics for the new layer are isolated from primary-path funnel analytics

## Verification Plan

- unit or integration checks for prompt selection rules
- regression verification that dashboard and daily-growth still load when playful data is unavailable
- existing auth, onboarding, dashboard, and daily-growth core specs still pass

## Explicitly Out Of Scope

- visible redesigns
- partner inboxes
- reminders
- archives
- favorites collections
- extra prompt types beyond `Daily Spark` and `Favorite Us`
